D2L org hierarchy in Brightspace

D2L is the company behind Brightspace. In admin docs and the UI you will see Organization / Org unit — that means how courses and folders are nested inside the LMS, not MLRI's leadership or departmental structure.

This section is how LACE's content is arranged in Brightspace. Every course, program, and training area fits into this tree. Understanding it is the foundation for naming, versioning, user access, and reporting.

The Brightspace hierarchy

MLRI
└── Massachusetts
    ├── Legal Skills  (Training Area)
    │   ├── Intro to Legal Skills  (Program)
    │   │   └── Legal Writing Template  (Course Template)
    │   │       ├── ARCHIVED: Legal Writing V1  (Course Offering)
    │   │       └── Legal Writing V2  (Course Offering — current)
    │   └── Client Communication  (Program)
    │       └── ...
    └── Substantive Law  (Training Area)
        ├── Housing Law  (Program)
        │   └── Eviction Defense Basics Template  (Course Template)
        │       └── Eviction Defense Basics  (Course Offering — current)
        └── Benefits Law  (Program)
            └── ...

What Each Level Means

MLRI

The root org unit in MLRI's Brightspace site (the top of the D2L org tree). Everything else nests under this node. The name happens to be MLRI — it is still a platform container, not a diagram of MLRI internal teams.

Massachusetts

The current state/jurisdiction level. This layer exists to leave room for future expansion — other states, regions, partner organizations, or business units could be added here later without restructuring everything else.

Training Area

Broad content categories that help learners navigate and discover courses. Currently:

  • Legal Skills — Skills applicable across all practice areas
  • Substantive Law — Area-specific legal content

See Training Areas for full definitions.

Program

Groups related learning experiences together. A Program is a named curriculum or topic cluster, like Housing Law or Intro to Legal Skills. Learners may take multiple Course Offerings within a Program over time.

Course Template

A container that holds all versions of a course. For example, Legal Writing Template holds Legal Writing V1, V2, and any future revisions. Templates are not learner-facing — learners enroll in Course Offerings, not templates. Templates exist to keep version history organized.

Course Offering

The actual learner-facing course. This is where content, assessments, discussions, completion tracking, and activities live. When you enroll a learner, you enroll them in an Offering.

Section (optional)

A subdivision of a Course Offering. May be useful for different cohorts, organizations, or scheduled runs of the same course. Not required for every course — especially self-paced evergreen content.

Why this Brightspace layout

The key design decision here is supporting evergreen content — courses that stay live and get updated over time rather than being run once and archived. The hierarchy gives us:

  • A clean place for multiple versions of a course to coexist (Course Template)
  • A named home for related courses to be discovered together (Program)
  • Room to grow into other jurisdictions or partner orgs without restructuring (Massachusetts layer)
  • Separation between what learners see (Offerings) and how we manage versions (Templates)

Naming Conventions

Consistent naming keeps the Brightspace backend clean as the catalog grows. These rules apply to everything we create in Brightspace.

The Rules

Course Templates

Use the course name followed by "Template".

Legal Writing Template
Eviction Defense Basics Template
Client Intake Template

Templates are not learner-facing, so the name is purely administrative. The word "Template" makes it immediately obvious this is a container, not an enrollable course.

Current Course Offerings

Use a clean, readable name — no version number in the title unless needed for clarity. The "current" offering is just the clean name.

Legal Writing
Eviction Defense Basics
Client Intake Fundamentals
Why no version number on the current offering? Learners see this name. "Legal Writing" reads better than "Legal Writing V2." The version is tracked inside the template container.

Archived Course Offerings

Prefix with "ARCHIVED:" and include the version number.

ARCHIVED: Legal Writing V1
ARCHIVED: Eviction Defense Basics V1

Archived offerings should also be hidden from learners so they don't accidentally enroll. The ARCHIVED prefix makes them easy to spot in admin views.

Programs

Plain descriptive name — no special prefix. Should be clear to a learner browsing a catalog.

Intro to Legal Skills
Housing Law
Benefits Law
Client Communication

Training Areas

Broad, stable category names. We currently have two:

Legal Skills
Substantive Law

See Training Areas for what belongs in each.

When to Increment the Version Number

The version number (V1, V2) tracks major revisions — not minor edits. See Versioning & Archiving for the full policy on when to create a new version vs. editing in place.

Training Areas

Training Areas are the top-level content categories in the LACE Brightspace catalog. They sit below the Massachusetts level and above Programs. Every Program belongs to a Training Area.

Training Areas serve two purposes:

  1. Learner navigation — They help attorneys find relevant courses when browsing the catalog or using the Discover page.
  2. Administrative organization — They group related Programs for reporting and management.

Current Training Areas

Legal Skills

Courses that teach skills applicable across all practice areas — skills every attorney needs regardless of what kind of law they practice.

What belongs here:

  • Legal writing
  • Client communication and intake
  • Legal research methods
  • Courtroom and advocacy skills
  • Document drafting
  • Professional responsibility and ethics
  • Case management workflows

What does NOT belong here:

  • Anything specific to a substantive area of law (housing, benefits, immigration, etc.) — those go in Substantive Law

Substantive Law

Courses about specific areas of law — the rules, statutes, regulations, and procedures for a particular legal topic.

What belongs here:

  • Housing law (eviction defense, landlord-tenant)
  • Benefits law (SNAP, MassHealth, SSI, etc.)
  • Immigration
  • Family law
  • Consumer protection
  • Any other area-specific legal content

What does NOT belong here:

  • General legal skills that would apply in any practice area — those go in Legal Skills

The "When in doubt" rule

Ask: Is this course useful to an attorney who doesn't practice in this specific area?

  • Yes → Legal Skills
  • No, it's specific to a practice area → Substantive Law

Future Training Areas

The Training Area layer was designed to grow. Future additions might include:

  • Organizational — internal LACE/MLRI policies, onboarding, HR-adjacent training
  • Technology — tools, systems, software training for staff
  • Partner Organizations — training shared with outside partners or coalitions

Adding a new Training Area is a significant decision. It changes catalog navigation and affects how learners find content. Document the decision in the Decision Log before creating a new one in Brightspace.

User Types & Access Matrix Awaiting Sign-off

Compliance critical. Wrong access decisions risk unauthorized practice of law (UPL): a non-attorney completing training that implies they're qualified to practice. This page must be reviewed and signed off by MLRI legal authority (ED, supervising attorney, or counsel) before pilot launch.

This matrix is the single source of truth for who can access what training and what their completion record says. It exists because "user type" is actually three separate questions that get confused into one:

  1. Brightspace role — what they can DO in the platform (Learner, Course Admin, etc.)
  2. License status — who they ARE (licensed attorney, paralegal, advocate, community member)
  3. Content access — which programs and training areas they can SEE and MARK COMPLETE

The risk is in column 3. A community member completing "How to file an emergency motion" with a certificate reading "qualified to file motions" creates a paper trail of someone holding themselves out as competent to practice law — even if MLRI never intended to imply that.

Versioning & Archiving Policy

Because LACE courses are evergreen — meaning learners may revisit them over time and content may change as laws or best practices evolve — we need a clear policy for when to update a course in place vs. when to create a new version.

This policy is not fully decided yet. The threshold between "minor" and "major" updates is still an open question. The framework below reflects our working understanding. When this is finalized, update this page and mark the decision in the Decision Log.

The Core Distinction

Update TypeWhat to do
MinorEdit the existing live course
MajorCreate a new Course Offering, archive the old one

Minor Updates — Edit in Place

A minor update is a change that does not affect the learning outcome or require learners to retake the course.

Examples of minor updates:

  • Fixing typos or grammatical errors
  • Updating broken links
  • Refreshing examples without changing the substance
  • Clarifying confusing wording
  • Adding a note or callout for context
  • Updating formatting or visuals

What to do: Make the edit directly in the live Course Offering. No version increment, no archive needed.

Major Updates — Create a New Version

A major update is a change that meaningfully affects what the course teaches — either because the law changed, the instructional approach changed, or the content was substantially replaced.

Examples of major updates:

  • A law, regulation, or court decision changes what attorneys need to know
  • Substantial content replacement (more than ~30–40% of a module rewritten)
  • New required material added that all learners should complete
  • New assessments or completion requirements
  • A fundamentally different teaching approach for the same topic

What to do:

  1. Create a new Course Offering under the same Course Template
  2. Name it with a clean title (current version gets the clean name, old gets ARCHIVED: prefix)
  3. Hide the old offering from learners so they can't accidentally enroll
  4. Decide whether existing enrolled learners need to take the new version (see Learner Update Notices)

Archiving Checklist

When creating a new version:

  • New offering created under the correct Course Template
  • Old offering renamed: ARCHIVED: [Name] V#
  • Old offering hidden from learner view
  • New offering is the clean current name
  • Decision documented in the Decision Log (what changed and why)
  • Learner notification decision made (see Learner Update Notices)

Why We Don't Delete Old Versions

Archived offerings are kept (hidden, not deleted) for several reasons:

  • Learners who completed the old version have a completion record tied to that offering
  • Compliance or grant reporting may require evidence of what content existed at a point in time
  • It lets us compare old and new content if questions arise

Learner Update Notices Policy

When LACE makes a major update to a course, some enrolled learners may need to know — especially if the change involves a law or regulation that affects their practice. This policy covers how we decide who needs to know, what we tell them, and how we reach them.

This policy is not fully decided yet. The notification mechanism is still an open question. Brightspace email may not be reliable enough for important legal updates. The custom frontend dashboard is the likely long-term solution, but it's not built yet (Phase 3). Document the final decision in the Decision Log.

Step 1 — Decide Who Needs to Know

Not every update requires notifying enrolled learners. Use this framework:

ScenarioAction
Minor update (typo, link fix, wording)No notification needed
Major update — law or regulation changedNotify affected learners
Major update — instructional overhaulNotify if completion of new version should be required
New offering added, old archivedNotify only if learners need to retake

Step 2 — Decide What Action to Ask For

When notifying learners, be clear about what (if anything) they need to do:

  • No action required — "We updated this course. Your completion record is unchanged."
  • Recommended review — "The law changed in this area. We recommend reviewing the updated content."
  • Required retake — "This course has been substantially revised. Please complete the new version by [date]."

Avoid vague notices. If the change is important enough to notify about, be specific.

Step 3 — How to Reach Learners

Option A: Brightspace Email (current only option)

Brightspace can send email notifications to enrolled learners. Limitations:

  • Email deliverability is not guaranteed
  • Many attorneys use shared or monitored inboxes
  • No confirmation that the learner saw the message
  • Not recommended for critical legal updates on its own

Use for: Low-stakes notices, reminders, optional recommendations.

Option B: Custom Frontend Dashboard (future — Phase 3)

The long-term solution is a notification or banner in the learner's custom dashboard that appears when they log in. This is much more visible than email and confirms the learner actually saw it.

Use for: All important legal update notices, required retakes.

Status: Not built yet. This is a Phase 3 deliverable. Until Phase 3 is complete, email is the only available channel.

Option C: Direct Outreach

For critical updates affecting a small, known group — a practice area team, a cohort — direct outreach by the training team (email, Slack, a meeting) may be the most reliable option.

Documenting Notifications

When you send a learner notification, record it in the Decision Log:

  • What course was updated
  • Why it was a major update
  • Who was notified and how
  • What action was requested
  • Date sent

Offboarding a User

When someone leaves MLRI or no longer needs Brightspace access:

  1. Preserve their completion records — Do NOT delete the account if they have completion history. Completion data is tied to the account.
  2. Deactivate, don't delete — Deactivate the account so they can no longer log in, but their records remain
  3. Remove from active course enrollments — Unenroll from courses they won't complete
  4. Document the date — Note when access was removed and why
Why not delete? Deleting a user account removes their completion history, which may be needed for reporting, compliance, or if the attorney later rejoins.

Glossary

Key terms used in Brightspace and in this wiki. When someone on the team uses one of these terms, this is what they mean in the LACE platform.

Brightspace Terms

Course Offering

The actual learner-facing course — where content, assessments, discussions, completion tracking, and learning activities live. When you enroll a learner, you enroll them in an Offering. A Course Template may contain multiple Offerings (different versions over time).

In LACE: This is what attorneys see and interact with. Name it cleanly — no internal jargon or version numbers visible to learners.

Course Template

An organizational container that holds all versions of a course. Templates are not learner-facing — learners don't enroll in templates, they enroll in Offerings inside a template. Templates exist to keep version history organized without cluttering the main catalog.

In LACE: Named [Course Name] Template (e.g., Legal Writing Template). Contains the current Offering and any ARCHIVED versions.

Section

An optional subdivision of a Course Offering. Can be used for different cohorts, scheduled runs, or organizations taking the same course. Not required — especially for self-paced evergreen content.

In LACE: Whether to use Sections vs. separate Offerings for cohorts is still an open decision.

Program

A named grouping of related Course Templates. For example, Housing Law is a Program that contains multiple course templates related to housing and eviction law.

In LACE: Programs live inside Training Areas. A learner might be "in the Housing Law program," meaning they're working through courses in that cluster.

Training Area

A broad content category that groups related Programs. Currently: Legal Skills and Substantive Law. Helps learners navigate the catalog.

In LACE: The layer below Massachusetts in the hierarchy, above Programs. See Training Areas for full definitions.

Org (Organization — D2L term)

In Brightspace, Organization is D2L's word for the top-level tenant (the whole site). Everything lives under that Org. At MLRI, that top-level org unit is named MLRI. Do not read "Org" here as MLRI's internal HR or management structure.

Discover

Brightspace's built-in course catalog/discovery page. Learners can browse and self-enroll in courses through Discover. Whether LACE uses the built-in Discover page or a custom frontend catalog is an open decision.

Completion

A record in Brightspace that a learner has finished a course (or met whatever completion criteria were set). Completions are tied to the Course Offering and the learner account — this is why we deactivate accounts rather than deleting them.

Enrollment

The connection between a learner account and a Course Offering. A learner can be enrolled manually by an admin, bulk-imported via CSV, or self-enrolled through Discover.

Vanity URL / Custom Domain

The custom web address learners use to reach the Brightspace site (e.g., learn.example.org instead of a generic D2L URL). Must be finalized during the rep setup window — week of June 15–19, 2026 for LACE — because it touches every learner-facing link, login screen, and certificate.

Community of Practice

A Brightspace space (course or course-like) that can grant different access to different users based on credentials or role. Potentially useful for shared resources, role-specific materials, or discussion spaces where lawyers, advocates, and community members see different content within the same container.

Intelligent Agents

Brightspace's automation tool for triggering actions (such as emails) based on learner activity, completion, inactivity, or missing work. Used together with email templates to handle reminders, nudges, and completion notifications without manual intervention.

LACE-Specific Terms

LACE Project

Legal Aid and Community Education — the training unit running this learning platform. LACE leads instruction in advocacy skills and key areas of law for the statewide legal aid community.

LACE Pilots

Two pilot rollouts feeding into the partial public launch in mid-January 2027: Pilot 1 in September 2026 with an internal cohort (Pre-Trial Skills); Pilot 2 in October 2026 with the broader legal aid community (internal + external lawyers).

Evergreen Course

A course designed to stay live and be updated over time, rather than being run once and closed. Most LACE courses are evergreen — attorneys may revisit them as the law changes or as they need a refresher.

Major Update

A change to a course that meaningfully affects what it teaches — typically triggered by a law change, substantial content replacement, or new completion requirements. Requires creating a new Course Offering and archiving the old one. See Versioning & Archiving.

Minor Update

A change to a course that does not affect the learning outcome — typo fixes, broken links, refreshed examples. Can be made directly in the live Offering without creating a new version.

Archived Offering

A Course Offering that has been superseded by a newer version. Named ARCHIVED: [Name] V# and hidden from learners. Preserved so completion records remain intact.

Unauthorized Practice of Law (UPL)

Non-licensed individuals holding themselves out as qualified to practice law. A compliance concern for any training platform serving non-attorneys. See User Types & Access Matrix for how LACE prevents UPL risk through role-based access and certificate language.

Quick Tasks

Fast answers for common Brightspace admin tasks. No background reading required.

Course Management

Create a new Course Offering

  1. Go to Admin Tools → Course Management
  2. Click Create a Course
  3. Set the Template to the correct Course Template (e.g., Legal Writing Template)
  4. Name it following the naming conventions
  5. Set the correct Program and Training Area
  6. Set availability dates if applicable
  7. Save

Archive an old Course Offering

  1. Go to the Course Offering you want to archive
  2. Rename it: ARCHIVED: [Name] V#
  3. Go to Course Settings → set Active to Inactive (or set an end date in the past)
  4. Hide it from the Discover page / catalog
  5. Record the archival in the Decision Log

Copy content from one Offering to another

Used when creating a new version from an existing course:

  1. Open the new (destination) Course Offering
  2. Go to Course Tools → Import / Export / Copy Components
  3. Choose Copy Components from Another Org Unit
  4. Select the source course
  5. Choose which components to copy (modules, assessments, etc.)
  6. Review and complete the copy

User Management

Add a single user

  1. Go to Admin Tools → Users
  2. Search for the person — they may already have an account
  3. If not, click Create a User
  4. Fill in name, email, username
  5. Assign the correct Role (Learner, Instructor, etc.)
  6. Save

Enroll a user in a course

  1. Go to the Course Offering
  2. Go to Classlist
  3. Click Add Participants → Add Existing Users
  4. Search for the user
  5. Assign their role in the course (usually Learner)
  6. Enroll

Deactivate a user (when someone leaves)

  1. Go to Admin Tools → Users
  2. Find the user
  3. Open their profile
  4. Set Is Active to No (do NOT delete — this preserves completion records)
  5. Note the date and reason

Content Updates

Make a minor edit to a live course

  1. Go to the Course Offering
  2. Open the module or topic to edit
  3. Make your change
  4. Save — the change is live immediately

No versioning or notification needed for minor edits (typos, links, wording). See Versioning & Archiving for what counts as minor vs. major.

Check if a learner completed a course

  1. Go to the Course Offering
  2. Go to Grades or Progress
  3. Search for the learner by name
  4. View their completion status and date

Finding Things

Find a course in the Brightspace org

Admin Tools → Course Management → search by name or course code

Find all courses in a Training Area or Program

Admin Tools → Course Management → filter by Org Unit Path

See who is enrolled in a course

Open the Course Offering → Classlist
