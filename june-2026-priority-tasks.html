June 19 Support Transition

Plain-English meaning: June 19, 2026 is not the product launch and it does not mean the fall pilot must be ready. It means the implementation period ends and LACE moves into standard support. After that, D2L help is more likely to come through help desk tickets, documentation, and community resources instead of a dedicated implementation manager who can shape architecture and do back-end setup with the team.

The practical goal is to lock the foundation before the implementation team steps away. LACE can still build content, manage users, refine Supabase data, and polish the Hub later. The risky items are the ones tied to tenant-level configuration, identity, permissions, domains, and hierarchy because they can be slow, disruptive, or impossible for a Super Admin to change alone after the transition.

Lock Before Standard Support

  • Vanity URL: finalize the learner-facing domain and D2L-managed SSL path before launch links, SSO, bookmarks, and API callbacks depend on it.
  • User Attributes: ask D2L to configure the custom attribute set, confirm the 50-attribute capacity, and verify which attributes can drive Learning Groups and Manager Dashboard reporting.
  • Production auth model: confirm the split between user-context OAuth and Service User client credentials, including JWKS URL, scopes, Org Unit visibility, and troubleshooting path for insufficient-scope errors.
  • Org hierarchy: audit the parent layers before automation relies on them; adding courses is easy later, but reshaping parent nodes can disturb reporting, affiliated org units, and routing assumptions.
  • Completion logic: confirm whether native Brightspace video triggers are sufficient or whether LACE must rely on the custom Architecture B/Supabase tracking layer for stronger compliance evidence.
  • Login and SSO: test the custom HTML login page constraints and resolve SSO deep-linking, target parameter, and learner return-path behavior while implementation support is still available.

Critical Milestones

  • Vanity URL / custom domain: finalize domain and ownership by June 15-19, 2026, then send the requested URL to D2L for DNS records.
  • Brightspace test site: ask D2L to launch the test instance early in June so API sync work happens outside the live catalog.
  • Production login model: use 3-legged OAuth for user-context calls and headless/server-to-server OAuth with a Service User for administrative data harvesting and background sync.
  • Service User: create the non-human Brightspace account, assign a least-privilege role, and enroll it in the Org Units needed for catalog sync before coding against the production flow.
  • User Attribute configuration: provide D2L the final Attribute Name and Description table, request the custom set, and confirm Learning Groups/Manager Dashboard mapping.
  • Hierarchy audit: confirm the MLRI > Massachusetts > Training Area > Program parent structure before routing and reports depend on it.
  • LTI 1.3 readiness: incorporate LTI Asset Processor support, anonymous marking readiness, and Context Copy Notice workflows from the March 2026 release notes.
  • Attributes/groups freeze: finalize User Attributes and Learning Groups before relying on enrollment automation or manager reporting.
  • Custom login test: upload a branded HTML login page sample and document any Brightspace constraints before the fall pilot build.

OAuth Configuration

  • Create a dedicated Brightspace Service User; do not use a personal admin account for integration sync.
  • Create a least-privilege role such as Hub Sync Service with read access to course content and user identity; add Brightspace Data Sets access only if bulk reporting is needed later.
  • Enroll the Service User in the Org Units it needs to see for catalog sync.
  • Register the production OAuth 2.0 app in Manage Extensibility using the headless/server-to-server pattern for data harvesting.
  • Associate the registered app with the Service User.
  • Host an HTTPS JWKS URL for the public signing keys Brightspace uses to verify JWT client assertions.
  • Verify required core:*:* scopes for the approved API routes.
  • Wait for D2L implementation specialist confirmation on Service User role setup, Org Unit inheritance/enrollment requirements, Client Credentials/JWKS requirements, API versions, and production scopes.
  • Store short-lived tokens in secure, HTTP-only cookies.

Vanity URL & SSL

  • Choose the desired learner-facing Brightspace vanity URL.
  • Send the desired vanity URL to D2L so they can submit the internal request and return DNS records.
  • Preferred path: let D2L manage the SSL certificate and renewals through AWS Certificate Manager; MLRI IT adds the D2L-provided validation DNS record.
  • Fallback path: use MLRI's existing wildcard certificate only if IT accepts sending D2L the certificate and private key and owning renewals.

Server-to-Server Implementation

  • Implement JWT client assertions with alg, aud, and kid in the header.
  • Include iss, sub, iat, exp, and one-time jti values in the JWT payload.
  • Keep assertion expiration short, ideally 60 seconds or less.
  • Request access tokens from https://auth.brightspace.com/core/connect/token with grant_type=client_credentials.
  • Use Authorization: Bearer {token} for API requests, over HTTPS only.

Security Checklist

  • Never store private keys or Client IDs in public repositories.
  • Use Next.js/Supabase environment variables for secrets.
  • Do not use refresh tokens with Client Credentials; request fresh access tokens with each valid assertion.
  • Rotate JWKS keys periodically.
  • Confirm the sync job points at the correct LMS API versions, such as LP 1.47 or later where required.
