Meeting Prep

Use this page during the call. The goal is not to prove every technical detail. The goal is to leave with the supported production pattern.

Must ask Call flow Leave with Closing line

Must Ask

These are the questions that decide the architecture.

  1. Auth triangle: How do OAuth scopes, Service User role permissions, and Org Unit enrollment work together?
  2. Org Unit inheritance: If the Service User has access at Massachusetts, can it see current and future course offerings below it?
  3. Catalog source: Should learning_items use live API, Data Hub/BDS, or both?
  4. Versioning: Which LP/LE versions does MLRI's tenant support through /versions/check, and which should production pin to?
  5. Retired courses: Which supported signal removes old courses from the Hub catalog: IsActive, Status, Deleted, Data Hub fields, or something else?
  6. Launch URLs: Should Learning Hub store Brightspace URLs, or generate them from IDs at runtime?
  7. Ownership: Who creates OAuth apps, Service Users, JWKS setup, production approvals, and key rotation?
  8. Audit/logging: Where does Service User activity appear, and what logs should LACE keep?

Call Flow

1. Context

Learning Hub is a learner-facing front door. Brightspace remains the LMS and official record.

2. What works

Local OAuth spike works. We can read metadata/content for course offering 6703.

3. Main decision

Confirm production auth, access, scopes, versions, and sync source before building more.

4. Keep focused

Do not let LTI, vanity URL, or reporting details take over unless D2L says they affect Phase 1.

Leave With

  • Recommended Phase 1 architecture: auth, scopes, role, Org Unit access, and sync source.
  • Minimum endpoint list for course metadata, content outline, enrollments, completions, users, and version checks.
  • How to diagnose a failed API call: scope problem, role problem, or Org Unit visibility problem.
  • Whether Data Hub/BDS has the fields needed for catalog cards, module outlines, completions, and timestamps.
  • Who owns sandbox and production setup steps.

Closing Line

Before we end, I want to make sure I understand the recommended production path: what stays in Brightspace, what LACE can safely cache in Supabase, what auth model we should use, how Service User access inherits, and how we detect retired courses. I’m trying to avoid building something that works in a sandbox but breaks when this becomes real.
