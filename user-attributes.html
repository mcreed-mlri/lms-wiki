Brightspace User Attributes

Planning notes for the attributes that will drive access, Learning Groups, Manager Dashboard, reporting, UPL guardrails, and jurisdiction-specific course routing.

Overview

Working rule: User Attributes should describe who the learner is for routing and reporting. They should not become a messy duplicate HR database.

Brightspace User Attributes can support Learning Groups, Manager Dashboard, bulk reporting, and automated enrollment rules. D2L's default labels are corporate or university-flavored, so LACE should rename and constrain them around our access matrix, legal-aid audience, jurisdictions, and UPL risk.

D2L constraints to remember: Brightspace supports up to 50 total attributes. Current planning note from implementation: no more than 10 attributes should be multi-value. Attribute values should stay short, and uploads are managed through Brightspace CSV import after users already exist.

Recommendation

Your proposed changes are right:

  • State should become jurisdiction. The platform may expand beyond Massachusetts, and jurisdiction is the legally meaningful concept.
  • Role Type should become user_type. This should map to the LACE access matrix, not a generic job title.
  • Hired Date should become onboarding_date. Many users will be outside MLRI, so "hired" is misleading.
  • Bar number belongs in the model, but use it carefully. It may support attorney verification, certificates, and CLE reporting, but it should not be the only source of professional-status truth.
  • UPL acknowledgement should be tracked. For non-lawyers, store the actual acknowledgement in a durable records system and mirror a date/status into Brightspace for routing/reporting.
Do not use free-text role labels for access control. A person's title can be "advocate," "paralegal," "case handler," or "coordinator," but the access decision needs a controlled user_type from the access matrix.

Discovery Access API Flow

Discovery rule: the Hub should never ask the browser to hide restricted courses after the fact. The API should return only the learning items the signed-in user is allowed to discover.
  1. Sign in: the learner authenticates through Brightspace OAuth, future SSO, or another approved identity provider.
  2. Resolve profile: the Hub matches the signed-in account to a stable learner record and loads controlled attributes such as user_type, jurisdiction, access_status, and UPL acknowledgement status.
  3. Build candidate catalog: the Hub loads active course, module, and path metadata from the discovery store. Brightspace remains the system of record for users, enrollments, course delivery, and launch access.
  4. Apply the access matrix server-side: the discovery API filters the candidate catalog against LACE eligibility rules before search ranking, facets, recommendations, paths, or launch URLs are returned.
  5. Render allowed results only: the frontend receives an already-filtered response. Audience facets remain browse filters, not security controls.
  6. Launch into Brightspace: Brightspace still enforces enrollment and delivery permissions when the learner opens an allowed course.

Recommended data model: each learning item should carry eligibility metadata such as allowed user types, jurisdiction scope, UPL requirement, and attorney-only flag. The discovery endpoint should treat those fields as policy inputs, not as optional display labels.

Attribute Access Map

These are the minimum user attributes that should participate in discovery access. They should come from controlled values, not free-text titles.

AttributeAccess MeaningDiscovery Use
user_typePrimary access matrix category: attorney, paralegal, non_lawyer_advocate, admin, or another approved type.First gate for attorney-only, advocate-safe, admin-only, and public/partner content.
access_statusWhether the learner is approved, pending, suspended, or inactive.Only approved users should receive discovery results. Pending, suspended, and inactive users should receive no catalog or a blocked-state response.

Policy note: audience labels such as "New attorneys" or "All staff" help users browse, but they are not the access control. Access should be calculated from the user attributes, the LACE access matrix, and item eligibility metadata.

Multi-Value Budget

Because multi-value attributes are limited, reserve them for fields where multiple values are expected and operationally useful.

Multi-value rule: spend the multi-value budget on legal routing, not corporate hierarchy. For LACE, jurisdiction and practice_area matter much more than Department, Division, Region, or Location.

UPL & Bar Records

UPL and bar status are governance records, not just convenience fields. Brightspace can carry routing metadata, but the source of truth for signed acknowledgements should be a durable form/database record that can be audited.

NeedRecommended HandlingBrightspace Attribute
Attorney identifierStore bar number if needed for certificates, verification, or reporting.bar_number
Admission stateTrack only if it affects routing, certificates, or reporting.bar_jurisdiction
Non-lawyer UPL acknowledgementCollect a signed acknowledgement outside Brightspace, then mirror the date/status into Brightspace.upl_acknowledged_date
Access decisionUse controlled access status values, not notes in free text.access_status
Legal guardrail: do not let bar_number alone decide course access. Pair it with user_type, jurisdiction, UPL acknowledgement where needed, and the LACE access matrix.

Upload Process

  1. Create or update the controlled value list for each attribute before upload.
  2. Make sure each user already exists in Brightspace.
  3. Download the Brightspace Bulk Attribute Upload template.
  4. Populate only approved attributes. Empty cells can remove existing values, so treat blanks carefully.
  5. Validate values before upload: user names, dates, controlled lists, and multi-value formatting.
  6. Upload through the Bulk Attribute Upload tab.
  7. Spot-check several users in Brightspace and confirm Learning Groups / Manager Dashboard behavior.
  8. Record the upload date, file name, and who approved the value set.
