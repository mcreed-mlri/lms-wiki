What this page covers

This is a planning document. Supabase is connected and one table (learning_items) is live in early form. The rest of the schema is designed but not yet built. Status labels on each section show where things stand.

The LACE Learning Hub uses Supabase as its custom data layer — the place we store data that Brightspace does not hold, or that we need to shape for our own experience. Brightspace stays the system of record for courses, enrollments, completions, and grades. Supabase fills the gaps: extended user profiles, catalog records synced from Brightspace, engagement events, and learner feedback.

What belongs where

The most important design question is not "what columns should a table have" but "does this even belong in Supabase or does Brightspace already own it?" Getting this boundary wrong means duplicating data, building sync logic you don't need, or storing something in the wrong system.

Rule of thumb: If Brightspace already tracks it authoritatively and you can read it via the API, don't store it in Supabase. Only sync what you need to shape or extend for the Learning Hub experience.
DataLives inWhy
Course title, description, org unit ID Brightspace (synced to learning_items) We need this in Supabase for search and catalog display without hitting the BS API on every page load
Enrollments Brightspace only (for now) BS is authoritative; read via API at request time. Only pull into Supabase if reporting latency becomes a real problem
Completions and grades Brightspace only (for now) Same reasoning. Don't duplicate unless there is a concrete reason
User identity / login Supabase Auth + profiles Supabase Auth handles login; profiles extends it with org, license status, and Learning Group
What a user searched for Supabase events Brightspace does not capture this; it is purely Learning Hub behavior
Content ratings and flags Supabase feedback Custom signal; Brightspace has no equivalent
Open question: We do not yet know the full scope of what the Brightspace API can return reliably for our use cases. Before building sync logic for enrollments or completions, confirm what the API gives us and whether a real-time read is fast enough. Current integration status is tracked on LACE Architecture.

learning_items Live (early)

The catalog table. One row per Brightspace course offering. Synced by the Learning Hub's server-side sync route, not written directly by users. This is what catalog search and discovery pages query instead of hitting the Brightspace API on every request.

ColumnTypeNotes

bs_org_unit_idbigintBrightspace Org Unit ID; the stable foreign key back to BS. Used as the upsert key on sync
titletextCourse title from BS; used in search and catalog display
descriptiontextCourse description from BS; nullable
urltextLearner-facing Brightspace URL for the course
topic_countintNumber of content topics synced from BS structure; nullable until we confirm the API gives it reliably
statustextactive, draft, or archived; controls catalog visibility
synced_attimestamptzLast time this row was refreshed from BS
created_attimestamptzAuto-set by Supabase on first insert

Access pattern

  • Read: any authenticated user (public catalog)
  • Write: service role only, from the server-side sync route
  • No browser code should ever insert or update a catalog row directly

profiles Proposed

Extended user data that Brightspace does not hold. One row per authenticated user, keyed to Supabase Auth's auth.users table. This is how the platform knows a user's organization, license status, and Learning Group assignment for routing and access control logic.

Design dependency: The Learning Group routing rules and user type matrix being worked out in User Attributes map directly onto columns in this table. Build both in parallel rather than treating them as separate problems.
ColumnTypeNotes
iduuidMatches auth.users.id exactly; not a separate key
bs_user_idbigintBrightspace user ID; links this profile to a BS learner record
emailtextDenormalized from Auth for convenience; keep it in sync with auth.users
orgtextOrganization name (e.g., "Greater Boston Legal Services")
license_statustextlicensed_attorney, paralegal, law_student, other_advocate
learning_grouptextRouting group assignment; drives what catalog content a user sees
roletextApp role: learner, manager, or admin
updated_attimestamptzLast profile update
created_attimestamptzAuto-set by Supabase

Access pattern

  • A user can read and update their own row only (RLS: auth.uid() = id)
  • Admins can read all rows via a service role or an explicit admin policy
  • role and learning_group should not be user-writable; set them via a controlled server-side function or the admin panel only

events Proposed

Custom engagement analytics. Every meaningful action a learner takes in the Learning Hub that Brightspace does not capture: searches, catalog clicks, navigation paths, time-on-page, drop-off points. This is what eventually feeds the program evaluation and equity reporting phases.

ColumnTypeNotes
iduuidAuto-generated
user_iduuidFK to profiles.id; nullable if you want anonymous event capture
event_typetextsearch, catalog_click, course_start, page_exit, etc.
item_iduuidFK to learning_items.id; nullable for non-course events like search
metadatajsonbFlexible payload: search query, time spent, clicked URL, etc. See the warning below before using this
created_attimestamptzWhen the event fired
Be careful with the metadata column. JSONB is flexible, but anything stored here carries the same privacy obligations as any other column. Do not put names, email addresses, or anything beyond the user's user_id in the event payload. Before writing events from the frontend, agree on a list of allowed event types and their allowed metadata keys, and enforce those in the server-side route.

Access pattern

  • Insert: any authenticated user, for their own user_id only
  • Read: own rows for learners; all rows for admin and program staff
  • Events are append-only; no update or delete from the browser

feedback Proposed

Learner-generated signals about content quality: ratings, flags, and short-form usefulness notes. This feeds the content effectiveness views in the program reporting phase.

ColumnTypeNotes
iduuidAuto-generated
user_iduuidFK to profiles.id
item_iduuidFK to learning_items.id
ratingint1–5 scale; nullable if the user only submits a flag
flagtextoutdated, unclear, not_relevant; nullable
notestextShort free-text comment; nullable. Keep this short-form, not a full review system
created_attimestamptzAuto-set

Access pattern

  • One row per user per item; enforce with a unique constraint on (user_id, item_id)
  • User can insert or update their own row; no delete (preserve the signal)
  • Aggregate views (average rating, flag counts) should be computed server-side rather than exposing raw rows to all users

Row-level security (RLS)

Supabase enables RLS by default, but it does nothing until you write policies. A table with RLS enabled and no policies returns zero rows to everyone, which will look like silent bugs in your frontend. The pattern to follow:

  1. Enable RLS on every table as soon as you create it, before adding any data.
  2. Write a SELECT policy for authenticated users first. For user-scoped tables use auth.uid() = user_id. For the catalog (learning_items) any logged-in user should see all rows, so the condition is just true for authenticated.
  3. Write INSERT and UPDATE policies separately. SELECT access and write access are different grants; do not assume one covers the other.
  4. Use the service role for admin operations like the Brightspace sync that writes to learning_items. The service role bypasses RLS, so it belongs only in server-side API routes, never in browser code.
  5. Test policies in the Supabase dashboard by running queries under a specific user role in the SQL editor before shipping any feature that touches new tables.
The service role key bypasses RLS entirely. It must never appear in frontend code or be committed to version control. It belongs only in server-side environment variables (.env.local, Vercel secrets, etc.). Treat it the same as a database root password.

Naming conventions

Consistent naming prevents confusion once you have multiple tables with foreign keys that cross between Supabase IDs and Brightspace IDs.

  • Table names: lowercase snake_case, plural. learning_items, not LearningItem or learning_item.
  • Column names: lowercase snake_case, singular. license_status, not licenseStatus.
  • Brightspace foreign keys: prefix with bs_ to make the source obvious. bs_org_unit_id, bs_user_id.
  • Timestamps: always timestamptz (timezone-aware), never plain timestamp. Name them created_at and updated_at consistently across every table.
  • Status and type values: use lowercase strings with underscores, not integers or abbreviations. active not 1; licensed_attorney not LA.
  • Boolean columns: prefix with is_ or has_. is_published, not published.
Changing names later is painful. RLS policies, API routes, and frontend queries all reference column names directly. Decide on the right name before data is in the table.

Brightspace sync

The sync relationship is one-directional: Brightspace is always the source; Supabase is a downstream copy of the pieces the Learning Hub needs. We never write back to Brightspace from Supabase.

QuestionAnswer
Who initiates sync? The Learning Hub's server-side sync route, triggered manually or on a schedule. Not the browser.
What happens when a course changes in BS? The sync route upserts the matching learning_items row, keying on bs_org_unit_id.
What if a course is archived in BS? Set status = 'archived' in learning_items rather than deleting the row. This preserves the event and feedback history for that course.
User identity sync? Store bs_user_id in profiles to link a Supabase user to their BS record. Do not sync the full BS user object.
Completions and enrollments? Read from the BS API at request time for now. Only pull into Supabase if performance or reporting requirements demand it later.
Do not build ahead of the sync. Course 6703 is the current validation target. Get one course syncing cleanly end-to-end before designing a bulk sync pipeline. The schema above is ready when you get there, but adding complexity before you need it will slow you down.

FERPA and privacy

Learning records for identifiable students are FERPA-protected. Any data in Supabase that can be tied back to a specific person's educational activity falls under this obligation. That includes the profiles, events, and feedback tables.

Core obligation: FERPA-protected records may not be disclosed to third parties without consent. Your Supabase project settings, API keys, and RLS policies are part of your compliance posture, not just a technical detail.

Things to stay conscious of

  • Collect minimum necessary data. If there is no clear use case for a column, do not add it. Columns are easy to add later; removing them after data exists in production is harder.
  • The events metadata column is a risk surface. It is easy to accidentally log a user's name, email, or full search query. Define allowed keys and enforce them in the API route before writing anything to the table.
  • Small groups break anonymity. A "team completion rate" for a team of two people effectively discloses individual records. Design any reporting views with minimum group-size thresholds when that phase comes.
  • The service role has full access. Treat the service role key with the same care as a database root password. Rotate it immediately if it is ever exposed.
  • Get legal sign-off before the profiles table goes live. The user types and access matrix in Team Wiki — Legal Sign-offs needs to be resolved before we start storing extended user attributes.
FERPA applies to students in educational institutions, not necessarily all users. Our platform will serve a mix of licensed attorneys, paralegals, law students, and other advocates. Until legal has reviewed this mix, treat all user learning records as protected.
