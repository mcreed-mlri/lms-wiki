Already documented — state, don't re-ask

  • JWT assertion fields: we know the header needs alg, kid and payload needs iss, sub, iat, exp, jti
  • JWKS URL: we will host HTTPS public keys for Service User headless OAuth
  • Scope wildcards: patterns like users:userdata:* are valid — still ask for the safest specific read-only pilot set
  • Rate monitoring: API Rate Logging exists — ask about production thresholds for our instance size

Critical questions — not in standard docs

  • Minimum Phase 1 route list: which exact endpoints and matching scopes should we validate for course metadata, content outline, enrollments, completions, users, and version checks?
  • Failure diagnosis: when 6703 fails, how do we tell whether the cause is missing OAuth scope, missing Brightspace role permission, or missing Org Unit enrollment?
  • Org Unit inheritance: enrolling the Service User at Massachusetts (Jurisdiction) — does it automatically grant read access to all current and future Course Offerings beneath it, or must we automate enrollment for every new course?
  • Catalog discovery source: live API vs Data Hub/BDS for bulk learning_items sync
  • Explicit versioning: which LP/LE versions does MLRI's tenant support today via /versions/check, and should production pin to the oldest fully supported contract or latest supported contract?
  • Retired offerings: should sync use IsActive, Course Updater Status, Deleted, Data Hub fields, archive UI terminology, or another D2L-supported signal?
  • Launch URLs: are course/topic URLs stable enough to store, or should the Hub generate official Brightspace links from OrgUnitId/topic identifiers at runtime?
  • Freshness target: what must be near-real-time for learner trust, and what can safely be nightly or hourly?
  • Secondary_Manager: included in whoami/User endpoints by default, or a specific version?

Stack glossary

Plain-English notes for the tech words that keep coming up. Not a computer science textbook, just enough to keep the meeting from turning into alphabet soup.

Learning Hub app stack

Next.js
The framework that powers the Learning Hub website. It serves pages to learners and runs server-side API routes that can safely talk to Brightspace and Supabase without exposing secrets in the browser.
React
A JavaScript library for building UI out of reusable pieces (cards, buttons, dashboards). Next.js is built on top of React.
Node.js
The runtime that runs JavaScript on a server instead of only in a browser. Next.js API routes and sync jobs run on Node.
npm & packages
npm is Node’s package manager — think “app store for code libraries.” A package is one installable library (listed in package.json). Examples: the Brightspace client helper, date formatting, UI components.
TypeScript
JavaScript with type labels so the editor catches mistakes early. Most of the Learning Hub codebase is TypeScript (.ts / .tsx files).
API route
A server endpoint inside Next.js (e.g. /api/me/dashboard). The browser calls our route; our route calls Brightspace or Supabase with credentials kept on the server.
Environment variables
Secret configuration (client IDs, database URLs, keys) stored outside source code, usually in a .env file locally and in the host’s settings in production.

Data & hosting

Supabase
Our hosted database and auth service. It stores LACE-owned data — catalog cache (learning_items), engagement signals, profile preferences — not the official training record.
PostgreSQL
The relational database engine under Supabase. Data lives in tables (rows and columns), like a spreadsheet with relationships.
Git & GitHub
Git tracks code changes over time. GitHub hosts the repo and runs deploy workflows. This wiki page is static HTML on GitHub Pages; the Learning Hub app itself deploys separately.
PWA
Progressive Web App — the Learning Hub can be “installed” to a phone home screen from the browser without an app store download.

Web & integration basics

HTML, CSS, JavaScript
HTML is page structure, CSS is styling, JavaScript is behavior. Course wrappers and this flashcards page are mostly HTML/CSS/JS; the Hub app adds React and Next.js on top.
JSON
The text format APIs usually speak — structured data with keys and values, e.g. {"title": "Intro to APIs"}.
REST API
A set of web URLs you can call to read or write data. Brightspace’s REST API is how the Hub requests course metadata, enrollments, and content outlines.
Brightspace (D2L)
Our LMS — the engine for courses, enrollments, completions, and certificates. D2L is the company; Brightspace is the product.
LMS
Learning Management System — software that hosts courses and keeps official training records. Brightspace is our LMS; the Learning Hub is the custom front door on top.
OAuth & SSO
OAuth is the standard way apps exchange login tokens without sharing passwords. SSO (single sign-on) means one login works across the Hub and Brightspace.
JWT & JWKS
A JWT is a signed token that proves “this request is from our app.” JWKS is the public URL where Brightspace fetches our signing keys to verify that token in headless sync.
Service User
A non-human Brightspace account used for background sync jobs — not Marlie’s personal login. It needs a role, OAuth app linkage, and Org Unit enrollment.
Org Unit
Brightspace’s hierarchy node — tenant, jurisdiction, department, course offering, etc. Enrollment at a parent level may (or may not) grant access to children; confirm with D2L.
LTI
Learning Tools Interoperability — a standard for embedding external tools inside Brightspace. Useful for in-course tools; our catalog sync path is mostly API + deep links instead.
Data Hub / BDS
Brightspace’s bulk export path for scheduled reporting datasets. Better for large historical pulls; the live API is better for near-real-time catalog sync.

Technical FAQ in human language

Reference companion to the flashcards. Synthesizes Phase 1 requirements from the May 2026 Developer Platform documentation and LACE project strategy — use for implementation, not as meeting small-talk.

1. Authentication & identity

What is our primary authentication model for backend sync?

For Phase 1 catalog sync, use the OAuth 2.0 Client Credentials Grant — a headless server-to-server flow with JWT client assertions, not user-interactive logins.

What is required for the JWKS setup?

Provide a valid HTTPS-secured JWKS URL where Brightspace retrieves your app's public keys to validate the JWT signature. Supported algorithms include RSA and EC (P-256, P-384, P-521).

How do we handle the Service User?

Create a dedicated Service User and associate it with the OAuth 2.0 app. Assign a least-privilege role and enroll the user in the Org Units it must sync — e.g. Massachusetts jurisdiction level. Confirm inheritance rules with D2L.

2. API endpoints & versioning

Which API versions should we use?

"Latest" works for spikes, but D2L should confirm which LP/LE versions MLRI's tenant supports today via /versions/check, and whether production should pin to the oldest fully supported contract or the latest supported contract. Explicit versioning prevents unexpected breakage during LMS updates.

What endpoints are required for catalog discovery?

  • LP (Learning Platform): Course Offering metadata — titles, codes, start/end dates.
  • LE (Learning Environment): Content Table of Contents — module and topic structure into Supabase.

How do we handle the Clean View in the wrapper?

Append ?d2l_body_type=3 to topic launch links. Display-only — strips native LMS chrome without affecting Architecture B tracking logic.

3. Scopes & permissions

Can we use wildcards in our scopes?

Yes. Testing may use core:*:*, but production should be granular. Wildcards like users:userdata:* are valid at registration; the token request should specify only required read scopes (e.g. content:toc:read).

What if we get an Insufficient Scopes error?

The error message may indicate whether a general fallback scope could work when a specific resource-based scope is not yet available for that route.

4. Data sync & performance

Why are we caching data in Supabase?

Supabase is the custom data layer that caches metadata so the Hub stays fast — the UI should not depend on a remote API call on every page load.

How do we monitor API usage?

Use API Rate Logging to watch production thresholds and keep background sync jobs inside LMS rate limits.

What is the Context Copy Notice?

If relevant, ask whether this is LTI-only or can help server-side discovery. If it is LTI-only, leave it out of Phase 1 so the meeting stays focused on API/Data Hub sync.

5. Technical bright lines

System of record

Brightspace remains the sole system of record for official PII, enrollments, and compliance-ready completion records.

Architecture B

All custom navigation links must point to official Brightspace topic URLs so the engine continues recording topic visits for compliance audits.

Deactivate, do not delete

For legal aid data persistence, user records and old course versions should be archived or deactivated, never deleted, to preserve historical completion data.

How the pieces fit together

This is the mental model: LACE is the front door, Brightspace is the official record, and Supabase is the helpful middle layer that keeps the Hub fast and usable. The API is for live operational reads. Data Hub/BDS is better for bigger scheduled reporting pulls.
