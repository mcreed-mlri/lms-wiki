Project North Star

Purpose: LACE strengthens legal aid by strengthening learning. The project should expand access to high-quality, practical, evidence-informed training for advocates across Massachusetts.

MLRI is building a shared learning infrastructure for the legal aid community. Brightspace provides the LMS foundation; the LACE frontend and course model should make training easier to discover, navigate, revisit, and use in real practice.

Learning Model

  • Practical, skills-based legal training.
  • Short multimedia learning experiences.
  • Self-paced and asynchronous participation.
  • Evidence-informed adult learning approaches.
  • Shared resources that attorneys and advocates can use in practice.
  • Long-term, readily available access to legal knowledge and practice tools.

Initial Pilot Topics

  • Legal Foundations.
  • Pre-Trial Skills.
  • Trial Skills.

Design implication: the platform should feel accessible, practice-centered, trustworthy, and built for legal aid work, not like a generic corporate LMS.

Legal Sign-offs

Core risk: non-lawyers must not receive access or completion language that suggests authority to practice law.

Routing input: audience rules should account for LegalServer fields where available, especially Zip Code and Legal Problem Code from the Coordinated Intake Handbook.

Course Content Model

Rule: LACE course content is custom HTML. Brightspace hosts, enrolls, tracks, and reports.
LayerWhat It DoesOwner
Custom HTML course files Primary learner experience: pages, activities, navigation, visuals, and interaction patterns. LACE build
Course wrapper Consistent LACE navigation and structure inside Brightspace. LACE build
Brightspace LMS infrastructure: hosting, enrollment, access, completion, notices, and records. Brightspace admins
Native Brightspace tools Optional support tools, used only when they are simpler than custom HTML. Case-by-case

Content Rule

  • Default: build course content as custom HTML files.
  • Use Brightspace-native tools only when they solve a specific need: quizzes, assignments, discussions, surveys, email automation, or LMS-tracked activities.
  • Do not treat Creator+ / H5P as the course authoring model. Those features are optional references, not the foundation.
  • Avoid sub-modules. Use document templates, accordions, and tabs instead; top-level modules should match the course schedule.

Completion rule: a video visit is insufficient for completion. Use at least one verification action such as a check-in, quiz, reflection, or manual mark-complete.

Technical implementation details live in Phase 0: In-Course HTML Wrappers. Optional Brightspace-native feature notes remain in Brightspace Native Tools Reference.

Solo Course Workflow

Operating model: one central course owner builds, uploads, and maintains a curated evergreen catalog. Brightspace is the deployment, access, tracking, and records layer; local Git is the source of truth.
1

Build Locally

Write course HTML, scripts, styles, and assets in the local repo. Test the course before anything touches Brightspace.

Storage: Local IDE / Git
2

Version Shared Wrapper

Keep LACE navigation, shared CSS, config loaders, logos, and brand assets in versioned shared folders.

Storage: Organization / Public Files
3

Upload Course Content

Upload course-specific pages, images, downloads, and media into the Course Offering's Manage Files area.

Storage: Course Manage Files
4

Create Trackable Topics

Create Content or Course Builder nodes that point to each learner-trackable HTML file.

Storage: Brightspace Content
5

Sync Metadata

Use the Brightspace API to cache course title, description, Org Unit ID, status, audience, and launch URLs.

Storage: Supabase learning_items
6

Reuse Later

Use the LOR only after repeated legal chunks become painful to maintain manually.

Storage: LOR, Phase 2+

Working Procedure

  1. Draft the course locally. Create one folder per course, keep each legal micro-module as a discrete HTML file, and use the Brightspace/LACE content style rule of Lato at 19px.
  2. Commit before upload. Treat Git as the personal history and rollback point. Brightspace should receive finished files, not become the editing workspace.
  3. Keep shared infrastructure versioned. Use shared wrapper paths such as lace-wrapper/v1/course-nav.js and lace-wrapper/v1/course-style.css. Do not casually overwrite one global file that every live course depends on.
  4. Upload course-specific content to Manage Files. Keep predictable folders such as pages, images, downloads, and media inside the Course Offering.
  5. Create official Brightspace topics. Every page that should count for progress, completion, reporting, or resume links should exist as a real Content topic.
  6. Clean up both layers when removing content. If a file is deleted from Manage Files, remove or replace the matching Content/Course Builder topic too.
  7. Sync only approved metadata to Supabase. Store the Course Offering Org Unit ID, course title, description, status, category, audience, and Brightspace launch/resume URL in learning_items.
  8. Use clean-view launch links. Append d2l_body_type=3 to Brightspace topic links so LACE navigation can own the learner viewport while Brightspace still records activity.

Growth Rule: One Course Family, Many Jurisdiction Offerings

Do not build one giant all-state course. As LACE expands, each state should have its own published Course Offering when law, procedure, forms, deadlines, or referral resources differ. Reuse shared source files where appropriate, but keep learner-facing legal guidance jurisdiction-specific.

Think of each course as three layers:

  • Core concept layer: reusable learning that applies across states, such as client interviewing, issue spotting, trauma-informed practice, litigation skills, legal aid values, and general workflow concepts.
  • State-specific law and procedure layer: statutes, court rules, deadlines, agency procedures, forms, local practice norms, referral resources, and certificate/CLE language.
  • Local implementation layer: partner access rules, local contacts, office workflows, manager/reporting needs, and required completion rules.

Versioning

ChangeActionExamples
Minor Edit the live Course Offering. Typos, broken links, formatting, clarified wording.
Major Create a new Offering; archive the old one. Law change, major rewrite, new required content, new completion rules.

Archive Checklist

  • New offering created under the same Course Template
  • Old offering renamed ARCHIVED: [Name] V#
  • Old offering hidden from learners
  • Learner notice decision made
  • Decision recorded in the Decision Log

Do not delete old versions. Completion history and reporting may depend on them.

Learner Notices

ScenarioNotify?Message
Minor editNoNo notice.
Law or regulation changedYesExplain what changed and whether review is required.
Major instructional overhaulMaybeNotify if learners should review or retake.
Old offering archivedMaybeNotify only if learners must use the new version.

Notice Levels

  • No action: completion record is unchanged.
  • Recommended review: learner should review updated content.
  • Required retake: learner must complete the new version by a stated date.

For now: Brightspace email is the available channel. Critical updates may also need direct team outreach. Future dashboard banners are tracked in LACE Learning Hub.

User Management

RoleUse
LearnerTakes courses. No admin access.
Instructor / Course AuthorCreates or edits course content.
ManagerOptional reporting role for people who need to monitor assigned learner progress.
Course AdministratorManages enrollments, sections, and course settings.
Org AdministratorFull Brightspace access. Use sparingly.

Rules

  • Assign least privilege. Give the role needed for the job.
  • Plan attributes before automation. User Attributes can drive Learning Groups, enrollments, due dates, and reporting.
  • Confirm automation source. Decide which User Attributes come from SIS/LegalServer and which stay manual before freezing Learning Groups.
  • Use Learning Groups deliberately. Decide which groups are automatic and which require manual control.
  • Separate auth patterns. Use logged-in OAuth for user-context work; use a Service User/headless OAuth path for administrative sync and data harvesting.
  • Use Manager access only where reporting is needed. A learner cannot have two direct managers, though higher-level manager relationships are possible.
  • Enroll learners in Course Offerings. Do not enroll them in templates.
  • Archive, do not delete. Deleting can remove completion history or break reporting assumptions.
  • Document offboarding. Record date and reason.

Terminology Updates

  • Use Archive instead of Delete for Course Offering and Template operations.
  • Use View Archive / Archived Org Units in SOPs instead of Recycle Bin language.
  • Use Export instead of Archive for Data Purge actions.

Quick Tasks

Create Course Offering

  1. Admin Tools → Course Management → Create a Course
  2. Select the correct Course Template
  3. Name it using Naming
  4. Set Program, Training Area, and dates
  5. Save

Archive Offering

  1. Rename old Offering: ARCHIVED: [Name] V#
  2. Set Active to Inactive, or set an end date in the past
  3. Hide from Discover/catalog
  4. Record in the Decision Log

Copy Between Offerings

  1. Open destination Offering
  2. Course Tools → Import / Export / Copy Components
  3. Choose source course and components
  4. Review and complete copy

Add User

  1. Admin Tools → Users
  2. Search first
  3. Create user if needed
  4. Assign role
  5. Save

Enroll User

  1. Open Course Offering
  2. Classlist → Add Participants → Add Existing Users
  3. Search user
  4. Assign course role
  5. Enroll

Deactivate User

  1. Admin Tools → Users
  2. Open user profile
  3. Set Is Active to No
  4. Record date and reason

Minor Live Edit

  1. Open Course Offering
  2. Edit module or topic
  3. Save

Check Completion

Open Course Offering → Grades or Progress → search learner.

Find Course

Admin Tools → Course Management → search by name or course code.

Find by Area / Program

Admin Tools → Course Management → filter by Org Unit Path.

Who Is Enrolled

Open Course Offering → Classlist.
